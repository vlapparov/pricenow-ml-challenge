Task 1 a) - 8 pts¶

Whenever a user wants to board a mountain railroad in the ski resort, they have to present their keycard at a turnstile before being allowed to board. Every time a user scans their keycard at a turnstile, a frequency is created in the database.

Your task is now to find the 10 most frequent Valais Mountain Card users and the city they live in across all ski resorts. You can assume that all frequencies in the database belong to ski resorts that are part of the Valais Mountain Card. For this task, we define the most frequent users based on their total number of frequencies. If two users have the same amount of frequencies, they should have the same rank. Some accounts may have several addresses. In this case, you are free to choose which address to use.

Your output should include the following columns: rank, first_name, last_name, zip, city, nr_of_frequencies

Assuming the tables mentionned above are related in the following way: frequencies <dataCarrierId and permissionId > keycard <personId and id> person <accountId> address we could write the following query to get the ranking by the frequencies:

In [489]:
frequency_query = """
SELECT 
    "rank",
    "firstName",
    "lastName",
    zip,
    city,
    nr_of_frequencies
FROM
    (SELECT 
         p."firstName",
         p."lastName",
         a.zip,
         a.city,
         COUNT(*) AS nr_of_frequencies,
         RANK() OVER (ORDER BY COUNT(*) DESC) AS rank
     FROM 
         frequencies f
         JOIN keycard k ON f.data_carrier_id = k."dataCarrierId" and f.permission_id = k."permissionId"
         JOIN person p ON k."personId" = p.id
         JOIN address a ON p."accountId" = a."accountId"
     GROUP BY 
         p."firstName",
         p."lastName",
         a.zip,
         a.city
     ) subquery
WHERE 
    rank <= 10;
"""

Task 1 b) - 6 pts¶

Bluebird days not only offer excellent skiing conditions, they are also the days when you can charge higher ticket prices. Your task is to find the bluebird days for all of Pricenow's clients.

We define a bluebird day as the following:

  • at least 10cm of fresh snow during the previous (!!) day
  • at least 70% relative sunshine

In this task we assume the the column avg_precipitation correspond to the average amount of fresh snow. In this case we could write the following query to identify the bluebird days.

    plt.figure(figsize=(10, 5))

def fit_model(X_train, y_train, X_test, y_test, scale=False, linear_model=True):
    X_train, y_train = X_train.copy(), y_train.copy()
    X_test, y_test = X_test.copy(), y_test.copy()
    if scale:
        # Initialize StandardScaler object
        scaler = StandardScaler()

        # Fit and transform the feature set
        X_train = scaler.fit_transform(X_train)
        X_test = scaler.transform(X_test)

    if linear_model:
        print("Using LinearRegression model")
        model = LinearRegression()
    else:
        print("Using GradientBoostingRegressor model")
        model = GradientBoostingRegressor()

    model.fit(X_train, y_train)

    # Predict scores for testing data
    y_pred = model.predict(X_test)

    # Compute r2 score of predictions
    r2 = r2_score(y_test, y_pred)

    # Print model r2 score
    print("R-squared score:", r2)
    return model

Interpretation: R-squared is a statistical measure that represents the proportion of the variance in the dependent variable that is explained by the independent variables in a regression model.

In some cases, the model may perform so poorly that the variance in the dependent variable that is not explained by the independent variables is greater than the total variance in the dependent variable, resulting in a negative R-squared value. A negative R-squared indicates that the regression model fits worse than a horizontal line, which represents the mean of the dependent variable.

In our particular case, during the data analysis we see that the average scores before 2000 were much higher in average than the ones after 2003 while the feature set is containing features that do not tend to reflect this change: given music of the same genre before and after chosen dates, it would, most probably, have similar music related features (like key, danceability, etc.) as we see from the TSNE plot and exactly the same one-hot encoded genre features. The only feature that is time-related in this case- releaseyear wouldn't help because of our split: model is unable to learn the decreasing trend since no points after the dip were included in to the training set.

Thus we have experienced a concept drift after 2000s and the linear regression model fitted with data before that date could not be used for data after the 2003.

def gbm_cross_validation_with_params(
    X,
    y,
    parameters,
    n_folds=20,
):
    # Initialize KFold cross-validator
    kf = KFold(n_splits=n_folds, shuffle=True, random_state=42)
    results = []
    n_estimators, learning_rate = (
        parameters["n_estimators"],
        parameters["learning_rate"],
    )

    r2_scores = []
    # Loop through each fold in the KFold
    for train_index, test_index in kf.split(X):
        # Split the data into train and test sets
        X_train, X_test = X[train_index], X[test_index]
        y_train, y_test = y[train_index], y[test_index]

        # Fit the Gradient Boosting Regressor with the current hyperparameters
        gb_regressor = GradientBoostingRegressor(
            n_estimators=n_estimators, learning_rate=learning_rate
        )
        gb_regressor.fit(X_train, y_train)

        # Evaluate the model using R2 score and append to list of scores
        y_pred = gb_regressor.predict(X_test)
        r2_scores.append(r2_score(y_test, y_pred))

    # Calculate the mean and 90% CI for the R2 scores
    mean_r2 = np.mean(r2_scores)
    ci = 1.645 * np.std(r2_scores) / np.sqrt(n_folds)
    lower_ci = mean_r2 - ci
    upper_ci = mean_r2 + ci

    # Add the results to the list of results
    results.append((n_estimators, learning_rate, mean_r2, lower_ci, upper_ci))

    # Convert the list of results to a pandas DataFrame and return it
    results_df = pd.DataFrame(
        results,
        columns=["n_estimators", "learning_rate", "mean_r2", "lower_ci", "upper_ci"],
    )
    return results_df

def plot_gbm_evaluation_report(df):
    # create figure and axis objects
    fig, ax = plt.subplots(figsize=(15, 5))

    # fill area between lower_ci and upper_ci
    ax.fill_between(df.index, df["lower_ci"], df["upper_ci"], alpha=0.5)

    # plot mean_r2 line
    ax.plot(df.index, df["mean_r2"], color="black")

    # set xticks and labels using n_estimators and learning_rate columns
    xticks = df.index
    xticklabels = [
        f"n_est = {n_estimators}\n\nlr = {learning_rate}"
        for n_estimators, learning_rate in zip(df["n_estimators"], df["learning_rate"])
    ]
    ax.set_xticks(xticks)
    ax.set_xticklabels(xticklabels)

    # set axis labels and title
    ax.set_xlabel("Hyperparameters")
    ax.set_ylabel("R2")
    ax.set_title("R2 vs. Hyperparameters")

    # Find the index of the row with the highest mean_r2 value
    best_index = df["mean_r2"].idxmax()

    # Get the n_estimators and learning_rate values for the best model
    best_n_estimators = df.loc[best_index, "n_estimators"]
    best_learning_rate = df.loc[best_index, "learning_rate"]

    # Add a label to the plot indicating the best mean_r2 value
    plt.annotate(
        "Best mean R2: {:.3f}\n({} n_estimators, {} learning_rate)".format(
            df.loc[best_index, "mean_r2"], best_n_estimators, best_learning_rate
        ),
        xy=(best_index, df.loc[best_index, "mean_r2"]),
        xytext=(best_index, df.loc[best_index, "mean_r2"] + 0.02),
        ha="center",
        arrowprops=dict(facecolor="black", arrowstyle="->"),
        bbox=dict(boxstyle="round,pad=0.5", fc="yellow", alpha=0.5),
    )
    return {"n_estimators": best_n_estimators, "learning_rate": best_learning_rate}

  • Setting up a monitoring system that would track model performance and alert in case of degradation. Tools like Grafana and Prometheus could be of help.

  • We could set up alerts, that would be sent to different channels like Slack, g-mail, SMS etc. It will help to remediate important issues ASAP. This is not the best way to improve one's healthy sleep though :)

  • In order to better understand why the model performed in a certain way we should store key features like model version, input/output information, and metadata in order to reproduce its behavior and debug.

  • It is usually very useful to back up data. This way we could reproduce the scenario for debugging and evaluation as well as use it for development purposes.

  • Monitoring could be set for incoming data as well. The data schemas could be used to check for data freshness and validation to prevent failures and alert on such important events as data drift.

  • We could also consider implementing a feedback loop to use insights from data analysis to improve the model and perform tasks like retraining, finetuning, feature engineering etc.

Here are some strategies to handle the versioning and rollback of the deployed model:

  • First of all, we need to set up the version control. It could be done using Git to manage code and model versions. Once a new update is pushed the version should be updated. It is important to align the team on the model versioning conventions to avoid problems. Controlling model versions helps to track the evolution of the project, check for back-ward incompatibilities, and perform a rollback if needed.

  • Split to microservices and containerize them. This way we are sure to manage dependencies and ease the deployment process. Also splitting into microservices usually helps to decouple tasks and unblock team members that are working on different features independently.

  • Implement a tests to ensure code quality and prevent serious production bugs. This could be unit tests, integration tests, and end-to-end tests. It is also very important to conduct ML tests that aim to spot model regression. For this task, we would most probably need, production-like data and environment. Finally, unit tests should be automated and used to prevent buggy code deployed to production.

  • Use one or several deployment strategies to gracefully roll out an update and rollback in case of any problem. Canary, shadow, or green/blue methods could be chosen depending on the use case.

  • Establish a rollback strategy that would be used in case of any problem. Sometimes it might be tricky, especially when releasing an update including multiple microservices with different dependencies. To avoid such problems it is always better to release updates gracefully.
